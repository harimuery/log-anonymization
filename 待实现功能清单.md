## 一、P0 — 安全/合规硬缺口（必须补齐）

### ~~1.1 SecureLogger 编程式脱敏 API（Facade 模式）~~ ✅ 已实现

- **执行计划出处**：§7.3 "SecureLogger Facade（外观模式）"
- **实现文件**：`log-anonymization-core/src/main/java/com/example/anonymization/core/application/SecureLogger.java`
- **实现内容**：
  - `SecureLogger.mask(value, SensitiveDataType)` — 单字段脱敏（委派 `SensitiveToStringHelper.maskValue`，纳秒级）
  - `SecureLogger.mask(message)` — 整条消息走完整管道（BloomFilter → Detection → Masking → Audit）
  - `SecureLogger.mask(message, LogContext)` — 携带完整上下文的管道脱敏
  - `SecureLogger.safeToString(Object)` — 业务对象 `@SensitiveField` 反射脱敏
  - `SecureLogger.MASKER_MARKER` — SLF4J Marker 常量（`"MASKER"`），配合 TurboFilter 使用
  - `SecureLogger.getMaskingPort()` — 获取底层 `MaskingPort` 实例（高级场景）
  - `SecureLogger.init(MaskingPort)` / `SecureLogger.destroy()` — 生命周期管理
- **集成点**：`LogAnonymizationAutoConfiguration` 中 `maskingPort` Bean 创建后调用 `SecureLogger.init()`，容器关闭时通过 `SecureLoggerDestroyer` 调用 `destroy()`

### ~~1.2 Jasypt 配置加密集成~~ ✅ 已实现

- **执行计划出处**：§13.1 "安全设计原则"、§18.9 "技术选型总结"
- **实现文件**：`log-anonymization-spring-boot-starter/src/main/java/com/example/anonymization/starter/JasyptConfig.java`
- **实现内容**：
  - `JasyptConfig.createEncryptor(password)` — 工厂方法，创建 `PooledPBEStringEncryptor`
  - 算法：`PBEWithHMACSHA512AndAES_256`（NIST 推荐，AES-256 + HMAC-SHA512 密钥派生）
  - IV 生成：`RandomIvGenerator`（每次加密产生不同密文）
  - Salt 生成：`RandomSaltGenerator`（随机盐，防彩虹表）
  - 连接池：2 个并发加密器实例
  - 主密钥来源：环境变量 `JASYPT_ENCRYPTOR_PASSWORD` 或系统属性 `jasypt.encryptor.password`
- **集成点**：
  - `LogAnonymizationAutoConfiguration` 中注册 `@Bean("jasyptStringEncryptor")`，使用 `@ConditionalOnClass` + `@ConditionalOnMissingBean` 保证可选性
  - `LogAnonymizationProperties.SecretConfig` 新增 `jasyptEnabled` 开关（默认 `false`）
  - `secret.salt` 字段支持 `ENC(...)` 格式，由 Jasypt 自动解密
  - starter 的 `pom.xml` 中添加 `jasypt-spring-boot-starter` 为 `optional` 依赖

## 二、P1 — 功能完整性缺口（建议补齐）

### 2.1 CachePort 出站端口接口

- **执行计划出处**：§3.1 模块结构、§4.4 "出站端口"
- **缺失描述**：执行计划在 `api/port/` 包下定义了 `CachePort.java` 出站端口接口，用于抽象缓存操作。当前 `port/` 目录下**没有此文件**，`CaffeineCacheAdapter` 直接暴露 Caffeine 原生 Cache 对象，未通过 Port 接口隔离
- **影响**：Domain 层对 Caffeine 存在间接依赖，违反六边形架构"Domain 层零外部依赖"原则；无法替换为 Redis 等远端缓存实现
- **建议文件位置**：`log-anonymization-api/src/main/java/com/example/anonymization/api/port/CachePort.java`

### 2.2 WhitelistFilter 白名单过滤器 + DefaultWhitelistPatterns 默认白名单

- **执行计划出处**：§5.4 "白名单机制"
- **缺失描述**：执行计划定义了独立的 `WhitelistFilter` 类和 `DefaultWhitelistPatterns` 常量类，包含 4 种预编译的排除模式：
  - UUID 格式：`[0-9a-f]{8}-...-[0-9a-f]{12}`
  - 时间戳（毫秒级）：`1[3-9]\d{11}`
  - 标准流水号：`[A-Z]{2,6}\d{14,20}`
  - 版本号：`v?\d+\.\d+(\.\d+)?`
- **现状**：当前 `CompositeDetector` 内部有简单的白名单过滤逻辑，但：
  - 没有独立的 `WhitelistFilter` 类
  - 没有 `DefaultWhitelistPatterns` 预编译常量
  - 执行计划中提到的 `ResultAggregator.aggregate(whitelistFilter.filter(...))` 调用链未完整实现
- **影响**：UUID/traceId 等常见非敏感字段可能被误识别为敏感数据，增加不必要的正则匹配开销
- **建议文件位置**：
  - `log-anonymization-core/src/main/java/com/example/anonymization/core/infrastructure/filter/WhitelistFilter.java`
  - `log-anonymization-core/src/main/java/com/example/anonymization/core/infrastructure/filter/DefaultWhitelistPatterns.java`

### 2.3 Redis 分布式缓存适配器

- **执行计划出处**：§10.1 "多级缓存体系"、§10.2 "缓存配置代码"、§18.4 "Redis 缓存配置"
- **缺失描述**：执行计划定义了 L2 分布式缓存层（Redis），用于多实例共享规则缓存。当前仅实现了 L1 Caffeine 本地缓存，**没有 Redis 适配器实现**
- **影响**：多实例部署时，每个实例独立从 Nacos/本地文件加载规则，无法通过 Redis 共享缓存，规则变更时各实例刷新时间不一致
- **建议实现**：
  - 创建 `RedisCacheAdapter` 实现 `CachePort` 接口
  - 支持 `masking:rules:{appName}` Key 模式，TTL 60s
  - 在 `LogAnonymizationAutoConfiguration` 中按 `cache.redis.enabled` 条件装配
- **建议文件位置**：`log-anonymization-core/src/main/java/com/example/anonymization/core/infrastructure/cache/RedisCacheAdapter.java`

### 2.4 真正的 Disruptor RingBuffer 审计实现

- **执行计划出处**：§12.1 "异步审计架构"
- **缺失描述**：当前 `DisruptorAuditAdapter` 名为 "Disruptor" 但实际使用 `ScheduledExecutorService` + `ArrayList` 缓冲 + `synchronized` 同步，**并非真正的 LMAX Disruptor 无锁队列**。POM 中已声明 `disruptor 3.4.4` 依赖但未使用 Disruptor API
- **现状**：
  ```java
  // 当前实现（ScheduledExecutorService + ArrayList）
  private final List<AuditRecord> buffer;
  private final ScheduledExecutorService scheduler;
  public synchronized void record(AuditRecord record) { ... }
  ```
- **计划要求**：
  ```java
  // 执行计划要求（Disruptor RingBuffer）
  private final RingBuffer<AuditEvent> ringBuffer;
  private final Disruptor<AuditEvent> disruptor;
  // RingBuffer 大小 65536，ProducerType.MULTI，YieldingWaitStrategy
  ```
- **影响**：在高并发（50,000 条/秒）场景下，`synchronized` 可能成为性能瓶颈，Disruptor 的无锁设计可提供更高的吞吐量
- **建议**：重构 `DisruptorAuditAdapter` 为真正的 Disruptor 实现，或新建 `DisruptorAuditRecorder` 类

### 2.5 SpiExtensionLoader SPI 扩展加载器

- **执行计划出处**：§18.2 "Spring Boot AutoConfiguration — Composition Root"
- **缺失描述**：执行计划 AutoConfiguration 伪代码中定义了 `SpiExtensionLoader` Bean，用于加载 SPI 扩展实现（如从 `META-INF/services/` 加载自定义 Detector/Masker），但**代码中不存在此类**
- **影响**：当前仅支持通过 Spring Bean 注册扩展，无法通过 Java SPI 机制加载第三方 JAR 中的扩展实现
- **建议文件位置**：`log-anonymization-core/src/main/java/com/example/anonymization/core/infrastructure/spi/SpiExtensionLoader.java`

### 3.1 单元测试（全量）

- **执行计划出处**：§15 "测试策略（DDD 分层测试）"
- **缺失描述**：整个项目 `src/test/` 目录下**没有任何测试类**。执行计划要求 80% 单元测试覆盖率，按 DDD 分层：

- **影响**：无法保证代码质量，重构/升级时无回归保护

### 4.4 数据库迁移脚本（Flyway/Liquibase）

- **执行计划出处**：§9.2 "DDL 脚本"
- **缺失描述**：执行计划定义了完整的 3 张表 DDL（`masking_rule`、`masking_rule_scope`、`audit_log`）+ MySQL Event 自动分区脚本，但**没有 Flyway 或 Liquibase 迁移脚本**用于自动化数据库初始化

### 4.5 版本兼容与升级处理代码

- **执行计划出处**：§18.8 "版本兼容与升级策略"
- **缺失描述**：执行计划定义了 5 项版本兼容策略：
  - 规则格式版本化（YAML 头部 `version: 1`）
  - SPI 接口向后兼容（仅新增 default method）
  - 灰度升级（同一集群不同版本 SDK 共存）
  - 废弃策略（`@Deprecated` + 至少保留 2 个大版本）
  - 配置兼容（新增配置项提供合理默认值）
- **现状**：YAML 规则文件中有 `version: 1` 字段，但**代码中没有任何版本解析、兼容性判断、灰度路由等逻辑**
- **影响**：未来 SDK 升级时缺乏平滑过渡能力
