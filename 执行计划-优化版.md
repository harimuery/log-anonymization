| # | 优化项 | V2.0 问题 | V3.0 优化方案 |

### 2.2 限界上下文（Bounded Contexts）

| 限界上下文 | 核心概念 | 主要类 | 所在层 |
|-------------|----------|--------|--------|
| **Rule Context** | 脱敏规则聚合根、规则版本、作用域 | `MaskingRule`（聚合根）、`MaskingScope`、`RuleValidator` | Domain + Application |
| **Detection Context** | 敏感数据识别、检测策略、快速否定 | `SensitiveDataDetectionService`、`CompositeDetector`、`SensitiveDataBloomFilter` | Domain + Infrastructure |
| **Masking Context** | 脱敏算法执行、熔断降级、结果合并 | `SensitiveDataMaskingService`、`ResilientMaskingEngine`、`FallbackMasker` | Domain + Infrastructure |
| **Audit Context** | 审计记录、异步投递、合规报告 | `AuditRecord`、`DomainEventBus`、`DisruptorAuditAdapter` | Domain + Infrastructure |

### 2.3 DDD 分层依赖规则

```
依赖方向（始终由外向内，禁止反向依赖）:

  Adapter ──▶ Application ──▶ Domain ◀── Infrastructure
 (logback/      (pipeline/     (model/      (RE2J/AC/Caffeine/
  log4j2/        facade/        spi/          Disruptor/
  starter)       stages)        ports/        Nacos/Resilience4j)
                                events)

规则:
  1. Domain 层: 零外部依赖，纯 Java 17，可独立编译和测试
  2. Application 层: 仅依赖 Domain 层，编排领域服务和管道
  3. Infrastructure 层: 实现 Domain 定义的 Port 接口，引入第三方依赖
  4. Adapter 层: 调用 Application 层，桥接日志框架 API
```

### 4.2 聚合根 MaskingRule + Builder 模式

```java
// 文件位置: log-anonymization-api/src/main/java/com/example/anonymization/api/model/

/**
 * 聚合根：脱敏规则
 * 设计模式：Builder 模式（复杂对象构建）
 * 不变性约束：所有字段 final，构建后不可变，线程安全
 */
public final class MaskingRule {
    private final String ruleId;
    private final String name;
    private final SensitiveDataType dataType;
    private final DetectorType detectorType;
    private final MaskerType maskerType;
    private final DetectorConfig detectorConfig;
    private final MaskerConfig maskerConfig;
    private final MaskingScope scope;
    private final int priority;
    private final boolean enabled;
    private final int version;

    private MaskingRule(Builder builder) {
        this.ruleId = Objects.requireNonNull(builder.ruleId, "ruleId 不能为空");
        this.name = Objects.requireNonNull(builder.name, "name 不能为空");
        this.dataType = Objects.requireNonNull(builder.dataType, "dataType 不能为空");
        this.detectorType = Objects.requireNonNull(builder.detectorType);
        this.maskerType = Objects.requireNonNull(builder.maskerType);
        this.detectorConfig = Objects.requireNonNull(builder.detectorConfig);
        this.maskerConfig = Objects.requireNonNull(builder.maskerConfig);
        this.scope = builder.scope != null ? builder.scope : MaskingScope.global();
        this.priority = builder.priority;
        this.enabled = builder.enabled;
        this.version = builder.version;
    }

    // 领域行为：判断规则是否适用于当前上下文
    public boolean appliesTo(LogContext context) {
        return enabled && scope.matches(context);
    }

    // 领域行为：比较优先级
    public int comparePriority(MaskingRule other) {
        return Integer.compare(other.priority, this.priority); // 高优先级在前
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String ruleId, name;
        private SensitiveDataType dataType;
        private DetectorType detectorType;
        private MaskerType maskerType;
        private DetectorConfig detectorConfig;
        private MaskerConfig maskerConfig;
        private MaskingScope scope;
        private int priority = 0;
        private boolean enabled = true;
        private int version = 1;

        public Builder ruleId(String id) { this.ruleId = id; return this; }
        public Builder name(String name) { this.name = name; return this; }
        public Builder dataType(SensitiveDataType dt) { this.dataType = dt; return this; }
        public Builder detector(DetectorType type, DetectorConfig config) {
            this.detectorType = type; this.detectorConfig = config; return this;
        }
        public Builder masker(MaskerType type, MaskerConfig config) {
            this.maskerType = type; this.maskerConfig = config; return this; }
        public Builder scope(MaskingScope scope) { this.scope = scope; return this; }
        public Builder priority(int p) { this.priority = p; return this; }
        public Builder enabled(boolean e) { this.enabled = e; return this; }
        public Builder version(int v) { this.version = v; return this; }

        public MaskingRule build() { return new MaskingRule(this); }
    }
    // getters 省略...
}
```

### 4.5 Specification 模式（规则条件组合）

```java
// 文件位置: log-anonymization-api/src/main/java/com/example/anonymization/api/specification/

/**
 * Specification 模式 — 可组合的规则条件
 * 设计模式：Specification（规范模式）
 *
 * 使用场景：规则作用域匹配、规则冲突检测、规则条件组合
 * 示例：ruleSpec.and(scopeSpec).and(prioritySpec)
 */
public interface Specification<T> {
    boolean isSatisfiedBy(T candidate);

    default Specification<T> and(Specification<T> other) {
        return candidate -> this.isSatisfiedBy(candidate) && other.isSatisfiedBy(candidate);
    }
    default Specification<T> or(Specification<T> other) {
        return candidate -> this.isSatisfiedBy(candidate) || other.isSatisfiedBy(candidate);
    }
    default Specification<T> not() {
        return candidate -> !this.isSatisfiedBy(candidate);
    }
}

/**
 * 规则匹配规范 — 判断规则是否适用于当前日志上下文
 */
public class RuleSpecification implements Specification<LogContext> {
    private final MaskingScope scope;

    public RuleSpecification(MaskingScope scope) { this.scope = scope; }

    @Override
    public boolean isSatisfiedBy(LogContext context) {
        return scope.matches(context);
    }
}
```

### 5.2 识别引擎算法选型（V2.0 修正）

> **V2.0 关键修正**：原方案将 Aho-Corasick 与正则进行不公正对比。AC 只能做精确关键字匹配，正则做的是模式匹配，两者解决不同层面的问题。

| 算法 | 适用场景 | 时间复杂度 | 在本系统中的角色 |
|------|----------|-----------|-----------------|
| **RE2J** | 银行卡号、身份证、手机号、邮箱、IP 等**模式匹配** | O(n) 线性 | **主力引擎**，处理 80% 的识别任务 |
| **Aho-Corasick** | 字段名 `password`/`cardNo`、上下文关键字 `cardNo=` | O(n + m + z) | **辅助引擎**，仅用于精确关键字匹配 |
| **字段名映射表** | JSON 字段名 `{"password":"xxx"}` | O(1) HashMap | **辅助引擎**，字段名→类型直接映射 |

**真正的性能优化重点**：
1. **RE2J 替代 `java.util.regex`**：保证线性时间复杂度，彻底杜绝 ReDoS
2. **预编译 Pattern 缓存**：所有正则在启动时编译并缓存，运行时零编译开销
3. **短路匹配**：命中高优先级规则（如 CVV、密码）后跳过后续低优先级规则

### 5.3 识别策略优先级

| 优先级 | 策略 | 说明 |
|--------|------|------|
| 0 (最高) | Bloom Filter 快速否定 | 先判断消息中是否可能包含敏感关键字，不含则直接跳过 |
| 1 | 白名单匹配 | 排除 UUID/时间戳/流水号，防止误杀 |
| 2 | 字段名精确匹配 | JSON 字段名 `password`/`pan`/`cvv` 直接命中 |
| 3 | 关键字上下文匹配 | `cardNo=xxx`/`password:xxx` 模式（AC 多模式） |
| 4 | RE2J 正则+校验位 | 身份证号（含校验位）、卡号（Luhn 校验） |
| 5 (最低) | 指纹相似度 | 长数字串、疑似格式化敏感数据（仅告警，不自动脱敏） |

### 5.5 Bloom Filter 快速否定层（V2.0 新增）

```java
/**
 * Bloom Filter 快速否定 — 在正则匹配前做 O(1) 的快速判断
 * 
 * 设计目的：80%+ 的日志不包含敏感数据，通过 Bloom Filter 快速跳过，
 *           避免对每条日志都执行昂贵的正则匹配
 * 使用场景：高流量场景下的性能优化，显著降低 CPU 占用
 * 
 * 原理：将所有规则的 keywords/fieldNames 放入 Bloom Filter
 *       - 如果消息中不包含任何关键字 → 100% 确定无需正则匹配（直接返回）
 *       - 如果 Bloom Filter 命中 → 可能包含，进入正则匹配确认（允许误判，不允许漏判）
 */
public class SensitiveDataBloomFilter {
    private volatile BloomFilter<String> filter;

    /**
     * 规则变更时重建 Bloom Filter
     * 将所有规则的 keywords、fieldNames、context keywords 加入过滤器
     */
    public void rebuild(List<MaskingRule> rules) {
        BloomFilter<String> newFilter = BloomFilter.create(
            Funnels.stringFunnel(Charset.UTF_8),
            expectedInsertions(rules),  // 预估插入量
            0.01                         // 1% 误判率（宁可多做正则，不可漏判）
        );
        rules.forEach(rule -> rule.getDetectorConfig().getKeywords().forEach(newFilter::put));
        rules.forEach(rule -> rule.getDetectorConfig().getFieldNames().forEach(newFilter::put));
        // 原子替换
        this.filter = newFilter;
    }

    /**
     * 快速判断消息是否可能包含敏感数据
     * @return false = 100% 不含敏感关键字，可跳过后续匹配
     *         true  = 可能包含，需进入正则匹配确认
     */
    public boolean mightContainSensitiveData(String message) {
        // 对消息做分段 token 化，检查每个 token 是否在 Bloom Filter 中
        // 使用滑动窗口而非全量分词，保证性能
        return checkTokens(message);
    }
}
```

### 5.6 识别器 Template Method + Composite + Chain of Responsibility（V3.0 新增）

> **V3.0 设计模式应用**：识别器采用 Template Method 提取公共逻辑，Composite 组合多种识别策略，Chain of Responsibility 编排管道阶段。

```java
// 文件位置: core/infrastructure/detector/

/**
 * 抽象正则识别器 — Template Method 模式
 * 设计模式：Template Method（模板方法）
 *
 * 子类仅需实现 doMatch() 和 getSupportedType()，公共逻辑由抽象类统一处理
 */
public abstract class AbstractRegexDetector implements SensitiveDataDetector {

    protected final List<Re2Pattern> compiledPatterns;

    protected AbstractRegexDetector(DetectorConfig config) {
        this.compiledPatterns = config.getPatterns().stream()
            .map(Re2Pattern::compile)
            .toList();
    }

    @Override
    public List<DetectionResult> detect(LogContext context) {
        String text = context.getMessage();
        List<DetectionResult> results = new ArrayList<>();

        for (Re2Pattern pattern : compiledPatterns) {
            Re2Matcher matcher = pattern.matcher(text);
            while (matcher.find()) {
                String matched = matcher.group();
                // Template Method 钩子：子类实现校验逻辑（如 Luhn 校验）
                if (validate(matched)) {
                    results.add(new DetectionResult(
                        matcher.start(), matcher.end(),
                        getSupportedType(), confidence()));
                }
            }
        }
        return results;
    }

    /** 钩子方法：子类实现特定校验（默认返回 true） */
    protected boolean validate(String matched) { return true; }

    /** 钩子方法：置信度（默认 1.0） */
    protected double confidence() { return 1.0; }
}

/**
 * RE2J 正则识别器 — 银行卡号（含 Luhn 校验钩子）
 */
public class Re2jBankCardDetector extends AbstractRegexDetector {
    @Override
    protected boolean validate(String matched) {
        return LuhnValidator.isValid(matched); // Luhn 校验位验证
    }
    @Override
    public SensitiveDataType getSupportedType() { return SensitiveDataType.BANK_CARD; }
}
```

```java
/**
 * 组合识别器 — Composite 模式
 * 设计模式：Composite（组合模式）
 *
 * 将多种识别器（RE2J、AC、字段名）组合为统一识别接口
 * 对外表现与单个识别器相同，内部委托给所有子识别器
 */
public class CompositeDetector implements SensitiveDataDetector {

    private final List<SensitiveDataDetector> detectors;
    private final WhitelistFilter whitelistFilter;

    public CompositeDetector(List<SensitiveDataDetector> detectors,
                             WhitelistFilter whitelistFilter) {
        this.detectors = List.copyOf(detectors);
        this.whitelistFilter = whitelistFilter;
    }

    @Override
    public List<DetectionResult> detect(LogContext context) {
        List<DetectionResult> allResults = new ArrayList<>();
        for (SensitiveDataDetector detector : detectors) {
            allResults.addAll(detector.detect(context));
        }
        // 白名单过滤 + 去重 + 优先级排序
        return ResultAggregator.aggregate(
            whitelistFilter.filter(allResults, context));
    }
}
```

```java
/**
 * 管道阶段接口 — Chain of Responsibility 模式
 * 设计模式：Chain of Responsibility（责任链）
 *
 * 每个阶段独立处理一个关注点，通过 next 指针串联成管道

    /** 处理上下文，决定是否传递给下一阶段 */

}

/**
 * 默认脱敏管道 — Chain of Responsibility 编排
 * 阶段顺序：BloomFilter → Detection → Masking → Audit
 */
public class DefaultMaskingPipeline {

    private final PipelineStage head;

    public DefaultMaskingPipeline(List<PipelineStage> stages) {
        // 链接各阶段
        for (int i = 0; i < stages.size() - 1; i++) {
            ((AbstractPipelineStage) stages.get(i))
                .setNext(stages.get(i + 1));
        }
        this.head = stages.get(0);
    }

    public MaskingResult execute(MaskingContext context) {
        head.process(context);
        return context.getResult();
    }
}

/**
 * Bloom Filter 快速否定阶段
 */
public class BloomFilterStage extends AbstractPipelineStage {
    private final SensitiveDataBloomFilter bloomFilter;

    @Override
    public void process(MaskingContext context) {
        if (!bloomFilter.mightContainSensitiveData(context.getMessage())) {
            context.setResult(MaskingResult.unchanged(context.getMessage()));
            return; // 短路：不传递到下一阶段
        }
        processNext(context); // 传递到下一阶段
    }
}
```

### 5.7 异常堆栈脱敏设计（V2.0 新增 — 填补原方案关键缺口）

> **背景**：异常堆栈是敏感数据泄露的重灾区。`SQLException` 的 message 中可能包含实际 SQL 参数值（含卡号、密码），连接异常可能暴露 IP 和密码。原方案仅在风险表中提及，缺乏详细设计。

```java
/**
 * 异常堆栈脱敏处理器
 * 
 * 设计目的：防止敏感数据通过异常 message、cause 链、suppressed 异常泄露到日志中
 * 使用场景：所有包含异常参数的 log.error()/log.warn() 调用
 * 
 * 处理范围：
 *   1. Throwable.getMessage() — 最常见泄露点
 *   2. Throwable.toString() — 包含类名+message
 *   3. Throwable.getCause() 链 — 递归遍历
 *   4. Throwable.getSuppressed() — 递归遍历
 *   5. 堆栈帧中的类名/方法名（通常不含敏感数据，但需扫描）
 */
public class ExceptionSanitizer {

    private final MaskingEngine maskingEngine;
    private static final int MAX_CAUSE_DEPTH = 10;  // 防止无限循环

    /**
     * 对异常链进行递归脱敏
     * 返回一个新的异常实例（不修改原始异常），其 message 已脱敏
     */
    public Throwable sanitize(Throwable throwable) {
        return sanitizeRecursive(throwable, 0, new IdentityHashMap<>());
    }

    private Throwable sanitizeRecursive(
            Throwable t, int depth, IdentityHashMap<Throwable, Throwable> visited) {
        if (t == null || depth > MAX_CAUSE_DEPTH || visited.containsKey(t)) {
            return t;
        }
        visited.put(t, t);  // 防止循环引用

        // 1. 脱敏 message
        String originalMessage = t.getMessage();
        String sanitizedMessage = maskingEngine.processMessage(originalMessage);

        // 2. 递归处理 cause
        Throwable sanitizedCause = sanitizeRecursive(t.getCause(), depth + 1, visited);

        // 3. 递归处理 suppressed
        Throwable[] suppressed = t.getSuppressed();
        Throwable[] sanitizedSuppressed = new Throwable[suppressed.length];
        for (int i = 0; i < suppressed.length; i++) {
            sanitizedSuppressed[i] = sanitizeRecursive(suppressed[i], depth + 1, visited);
        }

        // 4. 构建脱敏后的异常包装器（不修改原始异常）
        return new SanitizedThrowable(t, sanitizedMessage, sanitizedCause, sanitizedSuppressed);
    }

    /**
     * 异常包装器：保留原始异常的类名和堆栈，但 message 已脱敏
     */
    private static class SanitizedThrowable extends RuntimeException {
        private final Throwable original;
        private final String sanitizedMessage;

        SanitizedThrowable(Throwable original, String sanitizedMessage,
                          Throwable cause, Throwable[] suppressed) {
            super(sanitizedMessage, cause);
            this.original = original;
            this.sanitizedMessage = sanitizedMessage;
            for (Throwable s : suppressed) {
                addSuppressed(s);
            }
            // 保留原始堆栈
            setStackTrace(original.getStackTrace());
        }
    }
}
```

### 6.3 熔断降级 Fallback 设计（V2.0 优化）

> **V2.0 修正**：原方案所有类型使用同一占位符 `***MASKING_DEGRADED***`，导致日志无法区分数据类型。改为按类型差异化降级。

```java

 * 降级脱敏器 — 当熔断器打开时使用
 * 
 * 设计原则：
 *   1. 绝不输出原始明文（安全底线）
 *   2. 按数据类型提供差异化占位符（可排查性）
 *   3. 高危数据（CVV/密码）仍然执行丢弃/哈希

public class FallbackMasker implements SensitiveDataMasker {

    private static final Map<SensitiveDataType, String> FALLBACK_MAP = Map.of(
        SensitiveDataType.BANK_CARD,    "****DEGRADED****",
        SensitiveDataType.ID_CARD,      "***ID_DEGRADED***",
        SensitiveDataType.PHONE,        "***PHONE_DEGRADED***",
        SensitiveDataType.EMAIL,        "***EMAIL_DEGRADED***",
        SensitiveDataType.NAME,         "**NAME_DEGRADED**",

        SensitiveDataType.CVV,          "",                   // 仍然丢弃
        SensitiveDataType.IP_ADDRESS,   "***.***.***.***"
    );

    @Override
    public MaskingResult mask(String original, MaskerConfig config) {
        SensitiveDataType type = config.getDataType();
        String fallback = FALLBACK_MAP.getOrDefault(type, "***MASKED***");
        return MaskingResult.degraded(original, fallback);
    }

    @Override
    public MaskerType getMaskerType() { return MaskerType.FALLBACK; }

    @Override
    public boolean isReversible() { return false; }
}
```

```java
 * 公共逻辑（空值检查、配置校验、日志记录）由抽象类统一处理
 * 子类仅需实现 doMask() 钩子方法
 */
public abstract class AbstractMasker implements SensitiveDataMasker {

    @Override
    public MaskingResult mask(String original, MaskerConfig config) {
        // 公共逻辑：空值检查
        if (original == null || original.isEmpty()) {
            return MaskingResult.masked(original, original);
        }
        // 公共逻辑：配置校验
        validateConfig(config);
        // Template Method 钩子：子类实现具体脱敏逻辑
        String masked = doMask(original, config);
        return MaskingResult.masked(original, masked);
    }

    /** 钩子方法：子类实现具体脱敏逻辑 */
    protected abstract String doMask(String original, MaskerConfig config);

    /** 钩子方法：配置校验（默认无校验） */
    protected void validateConfig(MaskerConfig config) {}
}

/**
 * 部分遮盖脱敏器 — 模板方法具体实现
 */
public class PartialMaskMasker extends AbstractMasker {
    @Override
    protected String doMask(String original, MaskerConfig config) {
        int prefix = config.getKeepPrefixLen();
        int suffix = config.getKeepSuffixLen();
        char maskChar = config.getMaskChar();
        int maskLen = original.length() - prefix - suffix;
        return original.substring(0, prefix)
            + String.valueOf(maskChar).repeat(Math.max(0, maskLen))
            + original.substring(original.length() - suffix);
    }
    @Override
    public MaskerType getMaskerType() { return MaskerType.PARTIAL_MASK; }
    @Override
    public boolean isReversible() { return false; }
}
```

```java
/**
 * 脱敏器工厂 — Factory Method 模式
 * 设计模式：Factory Method（工厂方法）
 *
 * 根据 MaskerType 枚举值创建对应的脱敏器实例
 */
public class MaskerFactory {

    public MaskerFactory(List<SensitiveDataMasker> maskers) {
        maskers.forEach(m -> registry.put(m.getMaskerType(), m));

    public SensitiveDataMasker create(MaskerType type) {
        SensitiveDataMasker masker = registry.get(type);
        if (masker == null) {

        return masker;
```

```java
/**
 * 熔断装饰器 — Decorator 模式
 * 设计模式：Decorator（装饰器模式）
 *
 * 透明地包裹 MaskingApplicationService，在熔断器打开时自动降级
 * 调用方无感知，业务代码无需修改
 */
public class ResilientMaskingEngine implements MaskingPort {

    private final MaskingPort delegate;           // 被装饰的真实引擎
    private final CircuitBreaker circuitBreaker;   // 熔断器
    private final FallbackMasker fallbackMasker;   // 降级脱敏器

    public ResilientMaskingEngine(MaskingPort delegate,
                                  CircuitBreaker circuitBreaker,
                                  FallbackMasker fallbackMasker) {
        this.delegate = delegate;
        this.circuitBreaker = circuitBreaker;
        this.fallbackMasker = fallbackMasker;
    }

    @Override
    public MaskingResult process(String message, LogContext context) {
        return Try.of(() -> delegate.process(message, context))
            .recover(throwable -> {
                // 熔断器打开时，使用差异化 fallback
                return fallbackMasker.fallbackMask(message, context);
            })
            .get();
    }
}
```

### 8.4 启动时规则校验（V2.0 新增）

> **背景**：原方案缺少规则校验。坏规则可能导致 ReDoS、规则冲突、引用不存在的脱敏器等问题。

```java
/**
 * 规则校验器 — 在应用启动时和规则动态刷新时执行
 * 
 * 设计目的：防止坏规则导致服务不可用（ReDoS）或脱敏逻辑混乱
 * 使用场景：应用启动时加载规则、Nacos 推送规则变更时
 */
public class RuleValidator {

    /**
     * 校验单条规则，返回校验结果
     */
    public ValidationResult validate(MaskingRule rule) {
        List<String> errors = new ArrayList<>();
        List<String> warnings = new ArrayList<>();

        // 1. RE2J 兼容性检查：确保正则可被 RE2J 编译
        if (rule.getDetectorType() == DetectorType.REGEX) {
            for (String pattern : rule.getDetectorConfig().getPatterns()) {
                try {
                    Re2Pattern.compile(pattern);
                } catch (PatternSyntaxException e) {
                    errors.add("正则语法错误: " + pattern + " - " + e.getMessage());
                }
                // 2. ReDoS 风险评估：检测嵌套量词、指数级回溯模式
                if (hasNestedQuantifiers(pattern)) {
                    warnings.add("正则包含嵌套量词，可能存在 ReDoS 风险: " + pattern);
                }
            }
        }

        // 3. SPI 实现存在性校验
        if (!spiRegistry.hasDetector(rule.getDetectorType())) {
            errors.add("未找到识别器实现: " + rule.getDetectorType());
        }
        if (!spiRegistry.hasMasker(rule.getMaskerType())) {
            errors.add("未找到脱敏器实现: " + rule.getMaskerType());
        }

        // 4. 配置完整性校验
        if (rule.getPriority() < 0 || rule.getPriority() > 1000) {
            warnings.add("优先级值异常: " + rule.getPriority());
        }

        return new ValidationResult(rule.getRuleId(),
            errors.isEmpty(), errors, warnings);
    }

    /**
     * 批量校验 + 冲突检测
     */
    public List<ValidationResult> validateAll(List<MaskingRule> rules) {
        // 1. 逐条校验
        List<ValidationResult> results = rules.stream()
            .map(this::validate)
            .collect(Collectors.toList());

        // 2. 冲突检测：同一 fieldName 被多条规则覆盖
        detectFieldConflicts(rules, results);

        // 3. 启动时如有 ERROR 级别校验失败，阻止应用启动
        boolean hasErrors = results.stream().anyMatch(r -> !r.valid());
        if (hasErrors && failFastOnValidationError) {
            throw new RuleValidationException("规则校验失败，拒绝启动", results);
        }

        return results;
    }
}
```

### 11.2 大日志消息处理策略（V2.0 修复安全漏洞）

> **V2.0 关键修正**：原方案对超大消息直接截断尾部，尾部完全不扫描，导致敏感数据可能以明文输出。这是脱敏 SDK 不可接受的安全漏洞。

```java
/**
 * 大消息处理器
 * 
 * 设计目的：限制单条消息的最大扫描范围，防止超长消息导致 CPU 尖刺
 * 安全要求：截断部分的尾部也必须扫描，不能留安全死角
 */
public class LargeMessageHandler {
    private static final int MAX_SCAN_SIZE = 8192;    // 8KB 扫描上限
    private static final int MAX_MESSAGE_SIZE = 65536; // 64KB 消息上限

    private final MaskingEngine engine;

    public String handle(String message) {
        if (message == null) return null;
        int length = message.length();

        if (length > MAX_MESSAGE_SIZE) {
            // 超大消息：扫描头部 + 尾部各一半，中间截断
            int halfScan = MAX_SCAN_SIZE / 2;
            String headScanned = engine.processMessage(message.substring(0, halfScan));
            String tailScanned = engine.processMessage(
                message.substring(length - halfScan, length));
            int truncatedLen = length - MAX_SCAN_SIZE;
            return headScanned
                + "\n...[TRUNCATED:" + truncatedLen + "chars:NOT_SCANNED]...\n"
                + tailScanned;
        }

        if (length > MAX_SCAN_SIZE) {
            // 中等消息：扫描头部 + 尾部各一半
            int halfScan = MAX_SCAN_SIZE / 2;
            String headScanned = engine.processMessage(message.substring(0, halfScan));
            String tailScanned = engine.processMessage(
                message.substring(length - halfScan, length));
            int gapStart = halfScan;
            int gapEnd = length - halfScan;
            return headScanned
                + message.substring(gapStart, gapEnd)  // 中间部分原样保留（已标记未扫描）
                + " [LARGE_MSG_MIDDLE_UNSCANNED]"
                + tailScanned;
        }

        return engine.processMessage(message);
    }
}
```

### 11.3 采样脱敏策略（V2.0 新增）

> **背景**：50,000 条/秒 × 1ms = 50 核 CPU。在高流量场景下，需要对低风险数据类型支持采样脱敏以控制 CPU 占用。

```java
/**
 * 采样脱敏控制器
 * 
 * 设计目的：在高并发场景下，对低危数据类型按采样率脱敏，降低 CPU 占用
 * 安全底线：高危数据（CVV、密码、银行卡号）始终 100% 脱敏，不参与采样
 */
public class SamplingController {

    private volatile double samplingRate;  // 1.0 = 100%, 0.1 = 10%
    private final ThreadLocalRandom random = ThreadLocalRandom.current();

    /**
     * 判断当前日志是否需要脱敏
     * @param riskLevel 数据风险等级
     * @return true = 需要脱敏，false = 跳过
     */
    public boolean shouldMask(RiskLevel riskLevel) {
        // 高危数据始终脱敏
        if (riskLevel == RiskLevel.CRITICAL || riskLevel == RiskLevel.HIGH) {
            return true;
        }
        // 低危数据按采样率决定
        return random.nextDouble() < samplingRate;
    }
}
```

### 11.4 线程安全设计（V2.0 新增 — 填补原方案空白）

> **背景**：50,000 条/秒的并发压力下，共享状态的线程安全必须显式设计。原方案完全未涉及。

| 共享资源 | 并发风险 | 线程安全方案 |
|----------|----------|-------------|
| 规则列表 | 动态刷新时读写冲突 | `AtomicReference<List<MaskingRule>>` 原子引用交换 |
| AC 自动机 | 重建时并发读取 | Double-Buffer 策略：新建完成后原子替换引用 |
| Bloom Filter | 重建时并发读取 | 同上，`volatile` 引用替换 |
| RE2J Pattern 缓存 | 并发 put/get | Caffeine 内部线程安全，无需额外处理 |
| 指标计数器 | 高并发递增 | `LongAdder`（Micrometer Timer 内部已使用） |
| ThreadLocal | 线程池复用泄露 | 每次使用后 `ThreadLocal.remove()`，注册清理钩子 |

```java
/**
 * 线程安全的规则管理器
 * 
 * 核心设计：所有共享状态通过 AtomicReference / volatile 实现无锁读取
 * 规则刷新时：先在后台线程构建新的完整状态，完成后一次性原子替换
 */
public class ThreadSafeRuleManager {

    // 不可变规则列表引用 — 读取无需加锁
    private final AtomicReference<List<MaskingRule>> rulesRef =
        new AtomicReference<>(Collections.emptyList());

    // 不可变 AC 自动机引用 — double-buffer 策略
    private volatile AhoCorasickAutomaton acAutomaton;

    // 不可变 Bloom Filter 引用
    private volatile SensitiveDataBloomFilter bloomFilter;

    /**
     * 规则变更时调用 — 后台线程执行
     * 构建完整新状态后原子替换，读取线程无感知
     */
    public void refreshRules(List<MaskingRule> newRules) {
        // 1. 在后台线程构建新状态
        List<MaskingRule> sortedRules = sortByPriority(newRules);
        AhoCorasickAutomaton newAc = buildAutomaton(sortedRules);
        SensitiveDataBloomFilter newBf = buildBloomFilter(sortedRules);

        // 2. 原子替换（读取线程看到旧状态或新状态，不会看到中间状态）
        rulesRef.set(Collections.unmodifiableList(sortedRules));
        this.acAutomaton = newAc;
        this.bloomFilter = newBf;
    }

    /**
     * 热路径读取 — 无锁，直接读取 volatile/AtomicReference
     */
    public List<MaskingRule> getCurrentRules() {
        return rulesRef.get();
    }
}
```

```java
/**
 * 基于 Disruptor 的异步审计记录器
 * 
 * 设计目的：审计记录不影响脱敏主路径的延迟
 * 使用场景：每次脱敏处理后异步记录审计信息
 * 
 * 容量设计：
 *   - RingBuffer 大小 65536（2^16）
 *   - 单条审计记录 ~200 字节
 *   - 缓冲容量 ~12.8MB，足够应对突发
 *   - 批量刷盘：每 100 条或每 5 秒 flush 一次
 */
public class DisruptorAuditRecorder {
    private final RingBuffer<AuditEvent> ringBuffer;
    private final Disruptor<AuditEvent> disruptor;

    public DisruptorAuditRecorder(AuditExporter exporter) {
        this.disruptor = new Disruptor<>(
            AuditEvent::new,
            65536,  // RingBuffer 大小
            new ThreadFactoryBuilder().setNameFormat("audit-disruptor-%d").setDaemon(true).build(),
            ProducerType.MULTI,      // 多线程生产
            new YieldingWaitStrategy() // 低延迟等待策略
        );
        this.disruptor.handleEventsWithBatching(
            (events, lo, hi, endOfBatch) -> {
                List<AuditRecord> batch = new ArrayList<>((int)(hi - lo + 1));
                for (long i = lo; i <= hi; i++) {
                    batch.add(events[i].toRecord());
                }
                exporter.export(batch);
            }
        );
        this.disruptor.start();
        this.ringBuffer = disruptor.getRingBuffer();
    }

    /**
     * 异步投递审计记录 — 不阻塞调用线程
     */
    public void record(String traceId, String appName, String ruleId,
                       String dataType, String action, boolean success) {
        long sequence = ringBuffer.next();
        try {
            AuditEvent event = ringBuffer.get(sequence);
            event.set(traceId, appName, ruleId, dataType, action, success);
        } finally {
            ringBuffer.publish(sequence);
        }
    }
}
```

### 13.2 内存安全实践（V2.0 修正）

> **V2.0 关键修正**：原方案使用 `char[]` 清零是安全幻觉。日志消息在框架中已经是 String 对象，char[] 清零无法真正消除内存中的敏感数据。

```java
/**
 * 内存安全管理器
 * 
 * V2.0 设计理念：不追求不可能的 char[] 清零，而是关注真正可控的内存安全
 * 
 * 实际可控点：
 *   1. 脱敏后不持有原始消息的强引用（帮助 GC 尽早回收）
 *   2. ThreadLocal 使用后必须清理（防止线程池复用导致泄露）
 *   3. 审计日志中绝不记录原始明文
 *   4. 异常对象不缓存（避免长生命周期持有敏感异常信息）
 */
public class SecureMemoryManager {

    private static final ThreadLocal<LogContext> CONTEXT = new ThreadLocal<>();

    /**
     * 设置当前线程的日志上下文
     */
    public static void setContext(LogContext context) {
        CONTEXT.set(context);
    }

    /**
     * 获取并清理当前线程的日志上下文
     * 必须在 finally 块中调用，防止线程池复用时泄露
     */
    public static LogContext getAndClearContext() {
        LogContext ctx = CONTEXT.get();
        CONTEXT.remove();  // 立即清理，防止泄露
        return ctx;
    }

    /**
     * 验证审计记录中不包含原始敏感值
     */
    public static void validateAuditRecord(AuditRecord record, String originalValue) {
        // 确保审计记录中不包含原始明文
        if (record.errorMessage() != null
            && record.errorMessage().contains(originalValue)) {
            throw new SecurityException("审计记录中包含原始敏感值，违反安全策略");
        }
    }
}
```

### 13.3 日志注入防护（V2.0 新增）

> **背景**：OWASP Top 10 中的日志注入攻击 — 攻击者在输入中插入换行符，伪造日志条目。

```java
/**
 * 日志消息预清洗 — 在脱敏前执行
 * 清除消息中的换行符和控制字符，防止日志注入攻击
 */
public class LogMessageSanitizer {
    private static final Re2Pattern CONTROL_CHARS =
        Re2Pattern.compile("[\\x00-\\x08\\x0B\\x0C\\x0E-\\x1F\\x7F]");

    public static String sanitize(String message) {
        if (message == null) return null;
        // 替换控制字符（保留 \n \r \t，这些由日志框架自己处理）
        return CONTROL_CHARS.replaceAll(message, "");
    }
}
```

## 17. 风险点与应对策略

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| 正则 ReDoS 攻击 | 服务不可用 | 中 | 使用 RE2J 保证线性时间；启动时 RuleValidator 检测高危正则；运行时匹配超时 5ms 熔断 |
| 规则误杀业务数据 | 日志排查困难 | 高 | Bloom Filter + 白名单双重过滤；灰度发布新规则；规则测试工具；支持临时豁免 |
| 脱敏性能不达标 | TPS 下降 > 5% | 中 | Bloom Filter 快速否定（跳过 80%+ 日志）；RE2J 替代正则；采样脱敏；大消息头尾扫描 |
| 异常堆栈泄露敏感数据 | 合规风险 | 高 | ExceptionSanitizer 递归遍历 cause 链 + suppressed；异常 message 全量正则扫描 |
| toString() 泄露敏感字段 | 合规风险 | 高 | 第一层：Converter 正则扫描格式化消息；第二层：@SensitiveField + SensitiveToStringHelper |
| MDC 泄露敏感数据 | 合规风险 | 中 | AnonymizationMdcConverter 逐 key-value 独立检测脱敏 |
| 日志注入攻击 | 安全合规 | 中 | LogMessageSanitizer 预清洗控制字符和换行符 |
| 动态刷新导致规则不一致 | 部分节点规则未生效 | 低 | AtomicReference 原子切换；规则版本号校验；监控各节点规则版本 |
| 线程安全问题 | 数据竞争/脏读 | 中 | 全部共享状态使用 AtomicReference/volatile；ThreadLocal 使用后清理 |
| 内存泄露 | OOM | 低 | Caffeine 限制缓存大小；ThreadLocal 使用后清理；定期 GC 监控 |
| 第三方依赖冲突 | 应用启动失败 | 中 | 最小化依赖；BOM 统一管理；shade 隔离；可选依赖 optional |

```yaml
    flush-interval-seconds: 5

  # 熔断配置
  circuit-breaker:
    enabled: true
    failure-rate-threshold: 50
    slow-call-duration-threshold: 10ms

  # 密钥配置 [V2.0: v1 简化为配置注入，移除 KMS]

    salt: ${MASKING_SALT:default-salt-change-me-in-production}
```

### 18.2 Spring Boot AutoConfiguration — Composition Root（组装所有层依赖）

> **V3.0 DDD 角色**：Starter 的 AutoConfiguration 即 DDD 的 **Composition Root**，集中组装所有层的依赖关系。Domain 层的 Port 接口在这里绑定到 Infrastructure 层的 Adapter 实现。

```java
/**
 * Composition Root — 集中组装所有层依赖
 * DDD 角色：组装点，将 Port 接口绑定到 Adapter 实现
 * 设计模式：依赖注入 + Composition Root
 */
@AutoConfiguration
@ConditionalOnProperty(prefix = "log-anonymization", name = "enabled", havingValue = "true",
    matchIfMissing = true)
@EnableConfigurationProperties(LogAnonymizationProperties.class)
public class LogAnonymizationAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public MaskingEngine maskingEngine(
            List<SensitiveDataDetector> detectors,
            List<SensitiveDataMasker> maskers,
            CircuitBreakerRegistry circuitBreakerRegistry) {
        return new DefaultMaskingEngine(detectors, maskers, circuitBreakerRegistry);
    }

    @Bean
    @ConditionalOnMissingBean
    public RuleValidator ruleValidator() {
        return new RuleValidator();
    }

    @Bean
    @ConditionalOnMissingBean
    public SensitiveDataBloomFilter bloomFilter() {
        return new SensitiveDataBloomFilter();
    }

    @Bean
    @ConditionalOnMissingBean
    public ExceptionSanitizer exceptionSanitizer(MaskingEngine engine) {
        return new ExceptionSanitizer(engine);
    }

    @Bean
    @ConditionalOnMissingBean
    public MaskingMetrics maskingMetrics(MeterRegistry registry) {
        return new MaskingMetrics(registry);
    }

    @Bean
    @ConditionalOnMissingBean
    public SecureLogger secureLogger(MaskingEngine engine) {
        return new SecureLogger(engine);
    }

    @Bean
    public SpiExtensionLoader spiExtensionLoader() {
        return new SpiExtensionLoader();
    }
}
```

> **V2.0 说明**：Spring Boot 3.x 使用 `@AutoConfiguration` 替代 `@Configuration`，使用 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 注册。

### 18.8 版本兼容与升级策略（V2.0 新增）

> **背景**：SDK 作为嵌入式组件，升级策略极其重要。原方案未涉及。

| 策略 | 具体措施 |
|------|----------|
| 规则格式版本化 | YAML 文件头部 `version: 1`，解析器根据版本号适配 |
| SPI 接口向后兼容 | SPI 接口仅新增方法（default method），不删除/修改已有方法签名 |
| 灰度升级 | 同一集群中允许不同版本 SDK 共存；规则文件版本化确保新老兼容 |
| 废弃策略 | `@Deprecated` 标注 + 至少保留 2 个大版本 + 迁移指南 |
| 配置兼容 | 新增配置项提供合理默认值，旧配置项标记 `@Deprecated` 但继续生效 |

## 19. 设计模式全景映射表

> 本项目系统性应用了 **13 种设计模式**，每种模式都有明确的代码位置和解决的具体问题。

| # | 设计模式 | 分类 | 代码位置 | 解决的问题 |
|---|---------|------|----------|------------|
| 1 | **Strategy** | 行为型 | `api/spi/` 接口 | 识别器、脱敏器、规则加载器、审计导出器可插拔替换 |
| 2 | **Chain of Responsibility** | 行为型 | `core/application/pipeline/` | 管道各阶段可插拔编排（BloomFilter→Detection→Masking→Audit） |
| 3 | **Template Method** | 行为型 | `core/infrastructure/detector/Abstract*`、`masker/Abstract*` | 识别器/脱敏器公共逻辑提取，子类仅实现钩子方法 |
| 4 | **Observer** | 行为型 | `api/event/DomainEventBus` | 审计记录、指标上报与主路径解耦，规则变更通知 |
| 5 | **Decorator** | 结构型 | `core/infrastructure/resilience/ResilientMaskingEngine` | 透明包裹引擎，熔断时自动降级，调用方无感知 |
| 6 | **Factory Method** | 创建型 | `core/infrastructure/masker/MaskerFactory` | 根据枚举类型创建对应脱敏器实例 |
| 7 | **Registry** | 结构型 | `core/domain/DetectorRegistry`、`MaskerRegistry` | Strategy 实现集中注册和查找 |
| 8 | **Builder** | 创建型 | `api/model/MaskingRule.Builder` | 复杂聚合根构建，必填字段编译期校验 |
| 9 | **Specification** | 行为型 | `api/specification/Specification` | 规则条件可组合（and/or/not），复用性强 |
| 10 | **Port/Adapter** | 架构型 | `api/port/` + `core/infrastructure/` | Domain 层零依赖，通过接口反转实现依赖倒置 |
| 11 | **Facade** | 结构型 | `core/application/MaskingApplicationService`、`SecureLogger` | 封装复杂子系统，提供简洁对外 API |
| 12 | **Composite** | 结构型 | `core/infrastructure/detector/CompositeDetector` | 多种识别器组合为统一接口，递归委托 |
| 13 | **Flyweight** | 结构型 | `core/infrastructure/cache/CaffeineCacheAdapter` | RE2J Pattern、规则对象等重量级对象缓存复用 |

### 19.1 扩展点总结

| 扩展点 | 设计模式 | 扩展方式 | 业务方场景 |
|--------|---------|---------|------------|
| 自定义识别器 | Strategy + Registry | 实现 `SensitiveDataDetector` + Spring Bean 注册 | 业务特有格式（如内部令牌） |
| 自定义脱敏器 | Strategy + Factory + Registry | 实现 `SensitiveDataMasker` + Spring Bean 注册 | 仿真替换、令牌化 |
| 自定义规则加载 | Strategy + Port/Adapter | 实现 `MaskingRuleLoader` 或 `RuleLoadPort` | 非 Nacos/YAML 规则源 |
| 自定义审计导出 | Strategy + Observer | 实现 `AuditExporter` + 订阅 `DomainEventBus` | 对接 SIEM 平台 |
| 自定义管道阶段 | Chain of Responsibility | 实现 `PipelineStage` + Spring Bean 注册 | 前置校验、采样控制 |
| 自定义规则条件 | Specification | 实现 `Specification<LogContext>` | 复杂业务条件组合 |
